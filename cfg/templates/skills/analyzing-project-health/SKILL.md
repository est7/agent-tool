# 项目健康度分析

通过 6 维度量化评估 + 多角色辩论，产出可操作的改进路线图。

## 工作流

```
扫描度量 → 角色辩论 → 优先级共识 → 路线图输出
```

### 阶段 1: 扫描度量

对项目执行 6 个维度的量化扫描，每个维度独立评分（0-100）。

详细评分标准见 [references/scoring-criteria.md](references/scoring-criteria.md)。

| 维度 | 关注点 | 关键指标 |
|------|--------|----------|
| **依赖健康** | 过期/漏洞/废弃 | CVE 数量、过期依赖占比 |
| **文档完整** | README/API 文档/ADR | 公共 API 文档覆盖率 |
| **测试覆盖** | 单元/集成/关键路径 | 覆盖率 %、关键路径缺失数 |
| **安全态势** | OWASP/敏感信息/认证 | 高危漏洞数 |
| **架构质量** | 循环依赖/分层/耦合 | 循环依赖数、违规数 |
| **代码质量** | 圈复杂度/重复/异味 | 复杂度 >10 的函数数、重复率 |

**评分方法**：基于实际代码扫描结果打分，不可凭感觉。能用工具量化的必须量化（如运行测试覆盖率工具、检查 `package.json` 过期依赖等）。

### 阶段 2: 多角色辩论

根据扫描结果中得分最低的 2-3 个维度，选择最相关的角色展开辩论。

#### 可用角色

| 角色 | 视角 | 关注点 |
|------|------|--------|
| **security** | 风险最小化 | 合规性、零信任、最坏情况 |
| **performance** | 数据驱动 | 效率、用户体验、指标优化 |
| **architect** | 长期可持续 | 演进设计、技术债务、可维护性 |
| **frontend** | 用户中心 | 可访问性、交互体验、响应式 |
| **backend** | 系统稳定 | 可靠性、可扩展性、数据一致性 |
| **mobile** | 平台约束 | 设备限制、离线能力、电量优化 |
| **analyzer** | 证据优先 | 根因分析、假设验证、系统思维 |

#### 辩论流程

1. **立场陈述** — 每个角色基于扫描数据独立表达：建议、成功指标、潜在风险
2. **交叉讨论** — 指出对方盲点，澄清权衡关系，补充被忽视的视角
3. **寻找妥协** — 分阶段实施策略、双赢方案、风险缓解
4. **综合结论** — 达成共识的改进方案、实施路线图、验收标准

#### 辩论原则

- **事实导向** — 引用扫描数据和官方文档，不靠直觉
- **量化评估** — 用具体数据支撑论点
- **建设性** — 目标是找更优方案，不是否定对方
- **透明假设** — 明确说明判断依据和不确定性

### 阶段 3: 优先级共识

**优先级公式**: `Priority = (Impact × Frequency) ÷ Fix Cost`

| 优先级 | 响应时间 | 说明 |
|--------|----------|------|
| **P0** | 立即修复 | 安全漏洞、生产阻塞 |
| **P1** | 本周内 | 高影响、低成本 |
| **P2** | 本月内 | 中等影响、合理成本 |
| **P3** | 本季度 | 低影响或高成本 |

对每个改进项计算 ROI：

```
投入: N 小时
收益: <量化指标改善>
3个月 ROI: X%
```

## 角色选择策略

根据项目类型自动选择辩论角色：

| 项目类型 | 推荐角色组合 |
|----------|-------------|
| Web 前端 | frontend + performance + architect |
| 后端服务 | backend + security + architect |
| 移动应用 | mobile + performance + security |
| 全栈项目 | architect + security + performance |
| 通用/不确定 | architect + analyzer |

当用户指定角色时，使用用户指定的角色组合。
