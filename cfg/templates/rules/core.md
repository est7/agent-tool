# Core Rules — 核心原则与工作流

---

## 1. 核心原则 (Core Principles)

### 1.1 优先级栈

按以下顺序处理冲突，高优先级规则覆盖低优先级：

1. **正确性** — 代码必须正确工作
2. **可维护性** — 代码必须易于理解和修改
3. **性能** — 在满足前两项的基础上优化性能
4. **简洁性** — 避免不必要的复杂度

### 1.2 简洁性定义

- 每个函数/类仅承担**单一职责**
- 函数长度 ≤30 行，嵌套层级 ≤3 层
- 可读命名优于聪明技巧
- 如果需要额外解释，说明实现仍然过于复杂

---

## 2. 工作流约定 (Workflow Defaults)

### 2.1 修改前必做

- 先**阅读**相关代码，理解现有设计
- 找到 **2-3 个相似实现**作为参考
- 确认**测试框架**和验证方式

### 2.2 修改时原则

- **先给方案，再给 diff** — 复杂改动先说明思路
- **小步提交** — 每次提交保持可编译、可验证

### 2.3 破坏性操作确认

以下操作需要**明确确认**后才能执行：

- 删除文件或大段代码
- 修改数据库 schema
- 执行 `git push`（特别是 main/master 分支）
- 安装/卸载依赖

### 2.4 任务备忘录 (Task Memo)

当任务需要结构化落地时，必须将内容记录到项目的 `docs/memo/` 目录下，而非仅在对话中输出。

#### 文件规范

- **路径**: `docs/memo/YYYYMMDD-任务名称.md`
- **命名示例**: `20260210-用户认证重构.md`
- **同日多份**: 同一天出现同名时追加序号，如 `20260210-用户认证重构-2.md`
- **目录不存在时自动创建**: 首次写入前确保 `docs/memo/` 存在

#### 触发条件

满足以下**任一**条件时，必须创建备忘录：

- 任务包含 **7 个及以上**实现步骤
- 用户明确要求**记录 / 落地 / 沉淀**（如"记录一下"、"落地到文档"、"整理成方案"）

#### 内容结构

根据任务性质灵活组织，参考模板见 [`cfg/templates/spec/memo-template.md`](../spec/memo-template.md)。

#### 使用原则

- **实时更新** — 步骤完成后及时勾选或补充结果
- **内容优先于格式** — 信息完整比模板完美更重要
- **必须有结论** — 每份备忘录必须包含 `## 结论与产出`，明确可交付结果
- **一事一文档** — 不同任务独立建文件，不混写

---

## 3. 代码风格 (Code Style)

- **注释语言**: 中文（简体）
- **命名**: 英文，遵循语言惯例（camelCase/snake_case）
- **Import**: 遵循项目既有的 import 顺序和分组

---

## 4. 测试要求 (Testing)

- **每次实现**必须提供测试或验证方式
- **测试覆盖**: 正常流程 + 边界条件 + 错误恢复
- **测试命名**: 清晰描述被测场景，例如 `test_login_with_invalid_password_returns_401`
- 缺失测试时，必须在交付中说明原因和补测计划
