# 多角色辩论 (Role Debate)

启动多视角讨论，让不同专业角色围绕主题展开辩论，评估权衡，得出平衡的解决方案。

## 使用方式

```
/role-debate security,performance JWT Token 过期时间设置
/role-debate architect,frontend,mobile 状态管理方案选择
```

## 辩论流程（四阶段）

### 阶段1: 立场陈述
每个角色独立表达专业观点：
- 基于标准和证据的建议
- 成功指标定义
- 潜在风险识别

### 阶段2: 交叉讨论
角色间建设性辩论：
- 指出对方观点的盲点
- 澄清权衡关系
- 补充被忽视的视角

### 阶段3: 寻找妥协
探索平衡方案：
- 分阶段实施策略
- 双赢解决方案
- 风险缓解措施

### 阶段4: 综合结论
输出最终建议：
- 达成共识的方案
- 实施路线图
- 成功指标和验收标准

## 可用角色

| 角色 | 视角 | 关注点 |
|------|------|--------|
| **security** | 风险最小化 | 合规性、最坏情况、零信任 |
| **performance** | 数据驱动 | 效率、用户体验、指标优化 |
| **architect** | 长期可持续 | 演进设计、技术债务、可维护性 |
| **frontend** | 用户中心 | 可访问性、交互体验、响应式 |
| **backend** | 系统稳定 | 可靠性、可扩展性、数据一致性 |
| **mobile** | 平台约束 | 设备限制、离线能力、电量优化 |
| **analyzer** | 证据优先 | 根因分析、假设验证、系统思维 |

## 辩论原则

1. **事实导向** - 引用官方文档、标准规范、真实案例
2. **量化评估** - 提供具体数据和指标
3. **建设性** - 目标是找到更优方案，而非否定对方
4. **透明假设** - 明确说明判断依据和潜在风险

## 适用场景

- 技术选型（框架、数据库、云服务）
- 架构设计（微服务 vs 单体、同步 vs 异步）
- 权衡决策（安全 vs 性能、体验 vs 成本）
- 跨平台策略（Web/iOS/Android 统一方案）
